# Fantasy AGE DM Assistant - System Design v1

## 1) Purpose

This document defines the v1 architecture and runtime behavior for a deterministic + LLM DM assistant, aligned to the approved requirements in work-process/requirements/game_requirements_v1.md.

Design goals:
- Deterministic mechanics engine is authoritative for all state-changing outcomes.
- LLM is constrained to narrative and guidance, grounded in retrieved rules.
- Every impactful action is traceable, explainable, and reversible via transaction-safe writes.

## 3) System Context

External actors:
- DM client (UI or CLI): issues commands, enters rolls, confirms gated actions.
- Rules corpus artifacts: chunks/entities generated by canonical pipeline.

Frontend channels:
- Web app frontend is the primary DM experience surface.
- Future CLI channel remains supported for automation and diagnostics.

Core services:
- Session Orchestrator
- Rules Retrieval Service
- Deterministic Resolver Engine
- SQLite Persistence Adapter
- LLM Narration Service
- Validation and Guardrail Layer

## 4) High-Level Architecture

```text
DM Command
  -> Session Orchestrator
     -> Intent Classifier (state-changing vs informational)
     -> Rules Retrieval Service
        -> Chunk Index
        -> Entity Index
     -> Guardrail Layer (evidence sufficiency + policy checks)
     -> Deterministic Resolver (if mechanical)
     -> Confirmation Gate (if impactful)
     -> Transaction Writer (state + event log)
     -> LLM Narration (optional)
  -> Response (result + citations + reasoning + next options)
```

## 5) Service Responsibilities

### 5.1 Session Orchestrator
Responsibilities:
- Accept normalized command payloads.
- Coordinate retrieval, resolution, confirmation, persistence, and response formatting.
- Enforce mode preconditions (for example, combat action requires active combat scene).

Primary requirements:
- FR-001, FR-005, FR-015

### 5.2 Rules Retrieval Service
Responsibilities:
- Run hybrid retrieval (keyword + semantic style scoring).
- Return ranked evidence with citation IDs and confidence metadata.
- Provide reason codes when retrieval is insufficient.

Primary requirements:
- FR-013, FR-014, FR-016

### 5.3 Guardrail Layer
Responsibilities:
- Determine evidence sufficiency threshold by action type.
- Apply DN-003 policy:
  - State-changing actions: block on insufficient evidence.
  - Informational actions: allow uncertain response with no mutation.
- Validate input schema and scene constraints.

Primary requirements:
- FR-016, NFR-001, NFR-004

### 5.4 Deterministic Resolver Engine
Responsibilities:
- Compute tests, attacks, damage, conditions, stunt legality, and progression checks.
- Produce formula trace payload for explainability.
- Return proposed state diff (not committed yet).

Primary requirements:
- FR-004, FR-006, FR-007, FR-008, FR-009, FR-010, FR-011, FR-012

### 5.5 Confirmation Gate
Responsibilities:
- Require explicit DM approval before committing impactful state changes.
- Gate actions such as defeat/death outcomes, quest completion, level-up changes.

Primary requirements:
- FR-015

### 5.6 State Store + Event Log
Responsibilities:
- Persist runtime state and immutable event records in SQLite.
- Support ACID transaction boundaries for resolve + commit operations.
- Preserve input/output/citation traceability per action.

Primary requirements:
- NFR-002, NFR-004

### 5.7 LLM Narration Service
Responsibilities:
- Generate prose summaries, options, and uncertainty wording.
- Use retrieval citations and deterministic outputs as hard context anchors.
- Never directly mutate game state.

Primary requirements:
- FR-013, FR-014, NFR-005, NFR-006

## 6) Action Processing Flows

### 6.1 State-Changing Action (Exploration Test, Combat, Progression)

1. Orchestrator receives command payload.
2. Retrieval service returns evidence and confidence.
3. Guardrail checks evidence sufficiency and policy.
4. If insufficient evidence, action is blocked and DM decision required.
5. If sufficient, deterministic resolver computes result and proposed state diff.
6. Confirmation gate runs if action is high impact.
7. Transaction writer commits state and event atomically.
8. LLM narration generates optional explanation text from committed output.
9. Response returns result, citations, and audit identifiers.

Requirements covered:
- FR-004, FR-006, FR-007, FR-010, FR-016, FR-015, NFR-002, NFR-004

### 6.2 Informational Rules Query

1. Orchestrator classifies as non-state-changing.
2. Retrieval attempts evidence set.
3. If strong evidence, grounded answer returned with citations.
4. If weak/insufficient evidence, uncertain answer returned with explicit warning and no mutation.
5. Event log captures query and uncertainty status for audit.

Requirements covered:
- FR-013, FR-014, FR-016, NFR-002

## 7) Deterministic Resolution Modules

Module boundaries:
- test_resolver: ability/focus/modifier tests and margins.
- combat_resolver: hit logic, damage, health transitions, defeat states.
- stunt_resolver: SP generation and legal stunt validation by context.
- condition_resolver: apply/stack/tick/clear lifecycle.
- progression_resolver: level-up choices, prerequisites, derived stat recompute.

Resolver output contract:
- action_result
- formula_trace
- citations_used
- proposed_state_diff
- warnings

## 8) Retrieval and Evidence Policy

Evidence model:
- Each evidence item includes citation_id, source_type (chunk/entity), score, and matched fields.

Sufficiency thresholds (v1 defaults, configurable):
- state-changing:
  - minimum citation count >= 1
  - top evidence score >= 0.72
  - no unresolved direct conflict in top 3 evidence items
- informational:
  - minimum citation count >= 1 preferred
  - top evidence score >= 0.45 for normal grounded answer
  - if top score < 0.45, return uncertainty response with no mutation

Conflict rules:
- if top evidence items disagree on a mechanical rule that affects state, mark evidence_conflict = true.
- evidence_conflict for state-changing actions routes to blocked decision path.

Failure behavior:
- state-changing with insufficient evidence -> blocked, no state mutation, return decision prompt.
- informational with insufficient evidence -> uncertain response, no state mutation.

## 9) Data Persistence Strategy

Storage model:
- SQLite database as the canonical persistence store (single-file local DB in v1).
- Logical snapshot represented by latest session state rows keyed by session_id.
- Append-only event records stored in an events table for audit and replay.
- Optional checkpoint rows for accelerated replay on long-running sessions.

Write semantics:
- optimistic lock on session version.
- BEGIN IMMEDIATE transaction for commit pipeline; rollback on any failure.
- if write conflict, retry merge-safe operations; otherwise fail clearly and do not partially commit.

Traceability fields per event:
- input payload
- resolver output
- citations
- confirmation record
- transaction metadata

Suggested v1 SQLite tables:
- sessions(session_id PRIMARY KEY, state_version, scene_mode, round, turn_index, active_actor_id, payload_json, updated_at)
- session_entities(session_id, entity_type, entity_id, payload_json, updated_at)
- session_events(event_id PRIMARY KEY, session_id, pre_version, post_version, event_type, action_id, payload_json, citations_json, timestamp)
- confirmations(confirmation_id PRIMARY KEY, action_id, required, reason, status, dm_actor, decided_at)
- schema_migrations(version PRIMARY KEY, applied_at)

## 10) Error Handling

Error classes:
- validation_error: malformed command or missing required fields.
- precondition_error: wrong scene/mode for action.
- retrieval_insufficient_error: policy block for state-changing action.
- resolver_error: deterministic computation failure.
- persistence_error: failed commit/rollback path.

Response principle:
- never fail silently.
- always return actionable next steps for DM.

Canonical reason codes:
- validation.missing_field
- validation.invalid_enum
- precondition.scene_mismatch
- precondition.campaign_session_mismatch
- precondition.owner_scope_mismatch
- retrieval.insufficient_evidence
- retrieval.conflicting_evidence
- resolver.illegal_stunt_selection
- resolver.prerequisite_failed
- confirmation.rejected
- persistence.version_conflict
- persistence.transaction_failed
- guard.canon_warning_required
- guard.canon_delete_confirmation_required

## 11) Performance and Reliability Targets

- Deterministic mechanical resolution target: <= 150 ms locally, excluding LLM calls.
- Retrieval and resolver should be cache-aware for repeated session entities.
- All state mutations use transaction-safe commit path.
- If persistence fails, state remains unchanged and error is logged.

SQLite performance defaults:
- journal_mode = WAL
- synchronous = NORMAL
- busy_timeout = 5000 ms
- temp_store = MEMORY
- foreign_keys = ON

Expected local targets (p50/p95):
- read session snapshot: 5 ms / 20 ms
- append event + state update transaction: 10 ms / 40 ms
- replay 10,000 events with checkpoints: < 1.2 s p95

## 12) Security and Content Boundaries

- Use only local rules artifacts; avoid external unauthorized corpus pulls.
- Restrict stored data to session/game metadata and rule references.
- Preserve citation IDs instead of embedding large copyrighted passages in logs.

## 16) Runtime Command Contract

Incoming command envelope:
- action_id
- login_id
- campaign_id
- session_id
- actor_id
- command_type
- command_payload
- client_timestamp
- idempotency_key

Rules:
- idempotency_key is required for state-changing commands.
- duplicate idempotency_key for same session returns prior ActionResult without applying mutation twice.
- login_id is required for all commands in v1 (sourced from default login mode until auth exists).
- campaign_id and session_id must be validated as a bound pair before command execution.
- command_type maps to handler modules:
  - exploration.test
  - combat.attack
  - combat.apply_condition
  - progression.validate
  - progression.apply
  - rules.query
  - entity.create
  - entity.read
  - entity.list
  - entity.update
  - entity.delete
  - campaign.create
  - campaign.list
  - campaign.open
  - campaign.archive
  - session.create
  - session.list
  - session.open

Canon mutation guard:
- For `entity.update` and `entity.delete`, the orchestrator must resolve target entity provenance before execution.
- If target provenance is `canon`, warning acknowledgment is mandatory before mutation pipeline continues.
- Canon delete requires two-step gate: warning acknowledgment plus explicit confirmation.

## 17) Orchestrator State Machine

Pipeline states:
1. received
2. validated
3. evidence_loaded
4. policy_checked
5. resolved
6. confirmation_pending (optional)
7. committed
8. narrated
9. responded

Failure states:
- blocked_policy
- failed_validation
- failed_resolution
- failed_persistence

Invariant:
- committed can only occur after resolved and, when required, confirmation approved.
- committed cannot occur for canon update/delete unless canon warning gate is satisfied.

## 18) Deterministic Resolver Interface

Resolver interface (logical):
- input:
  - pre_state
  - normalized_command
  - retrieval_evidence
- output:
  - action_result
  - proposed_state_diff
  - formula_trace
  - warnings
  - required_confirmation (boolean)

Resolver constraints:
- resolvers are pure functions with no side effects.
- no direct database writes from resolver modules.
- every numeric output includes explicit source components in formula_trace.

## 19) SQLite Transaction Protocol

Commit algorithm:
1. BEGIN IMMEDIATE.
2. Read current session row and verify expected pre_version.
3. For entity.update/entity.delete, resolve target provenance and enforce canon warning gate.
4. Insert event row with pre_version and tentative post_version.
5. Upsert modified entities.
6. Update sessions row to post_version and payload snapshot.
7. Insert confirmation record when applicable.
8. COMMIT.

Rollback behavior:
- Any failure in steps 2-6 triggers ROLLBACK.
- API returns persistence error with reason code and retry guidance.

Idempotency behavior:
- Maintain command_receipts table keyed by (session_id, idempotency_key).
- On duplicate key, return stored action_result and correlation_id.

## 20) Schema Migration Strategy

Migration rules:
- migrations are forward-only files with semantic version labels.
- each migration runs in a transaction and writes to schema_migrations.
- app startup refuses to run if required migration level is not met.

Compatibility policy:
- additive schema changes are preferred in v1.x.
- destructive changes require export, migration tool, and restore verification.

## 21) Backup and Recovery

Backup plan:
- online SQLite backup API snapshot every N events (default 200) and at shutdown.
- daily timestamped file copy for disaster recovery.

Recovery plan:
- restore latest backup.
- verify integrity via PRAGMA integrity_check.
- replay missing events from replicated event export when available.

RTO and RPO targets (local deployment):
- RTO <= 10 minutes.
- RPO <= 200 events.

## 22) Observability and Alerting

Required metrics:
- command_total by command_type and outcome.
- resolver_latency_ms histogram by module.
- retrieval_score histogram by action category.
- blocked_actions_total by reason_code.
- db_transaction_latency_ms and db_conflict_total.

Structured logs per command:
- correlation_id
- action_id
- session_id
- command_type
- outcome
- reason_code (when not success)
- citation_ids

Alert conditions:
- p95 db_transaction_latency_ms > 100 ms for 15 minutes.
- blocked_actions_total spike > 2x baseline over 30 minutes.
- integrity_check failure at startup or scheduled run.

## 23) LLM Prompt Contract and Guardrails

Prompt inputs:
- deterministic action_result summary
- concise formula_trace
- top citations (ids + short snippets)
- response style directive (brief, neutral, DM-facing)

Guardrail rules:
- no state mutation instructions from LLM output.
- no uncited mechanical rulings if retrieval evidence is weak.
- uncertain informational responses must include missing evidence note.

Output envelope:
- narrative_text
- citations_used
- uncertainty_flag
- suggested_next_actions

## 24) Security, Privacy, and Retention

Storage boundaries:
- persist only session mechanics, metadata, and rule references.
- avoid storing unnecessary freeform personal data.

Retention defaults:
- event history retained for campaign duration.
- optional archival export after campaign closure.

Integrity protections:
- optional checksum per event payload.
- startup integrity verification and periodic consistency checks.

## 28) Multi-Campaign Architecture

Tenant model:
- campaign is the top-level gameplay container.
- session is a run within one campaign.

Isolation model:
- every command executes under a required (campaign_id, session_id) context.
- persistence queries and mutations are always scoped by campaign_id.
- event replay, audit views, and CRUD operations are campaign-bounded.

Context switching:
- UI supports active campaign selection and active session selection.
- switching campaign clears active session context and loads campaign-specific summaries.

Safety rules:
- if command context references a session outside the selected campaign, block with precondition error.
- idempotency keys are unique within a campaign/session boundary.

## 29) Login Ownership Scope (Pre-Auth Ready)

Owner model:
- every campaign, session, and runtime entity belongs to one login owner (`login_id`).
- all queries and mutations are filtered by active `login_id` context.

Default login mode:
- v1 uses a configurable default login id (for example `local-dev-dm`) when authentication is not yet implemented.
- default login id value is loaded from runtime config and stamped in audit metadata.

Safety rules:
- if command `login_id` does not own target campaign/session/entity, block with owner-scope precondition error.
- owner scope applies before campaign/session validation and before resolver execution.
